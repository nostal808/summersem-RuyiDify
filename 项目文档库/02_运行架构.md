## 2.1 组件总览

```
浏览器 ──HTTP──▶ [nginx :80/:443]  （docker 部署模式）
                    ├─▶ web  (Next.js，静态+SSR，:3000 容器内)
                    └─▶ api  (Flask/gunicorn :5001)
                           ├─▶ PostgreSQL / MySQL   (关系数据、账号、dataset 元数据)
                           ├─▶ Redis                 (缓存 + Celery broker/result)
                           ├─▶ 向量库 (Weaviate 等)   (embedding 向量)
                           ├─▶ 对象存储 (OpenDAL)     (上传文件、附件)
                           ├─▶ Plugin Daemon :5002    (模型/工具运行时)
                           ├─▶ DifySandbox :8194      (代码执行)
                           └─▶ SSRF Proxy :3128       (受限外网访问)
  [Celery worker] ──消费──▶ Redis 队列 (dataset/workflow/mail/…)
  [Celery beat]  ──周期──▶ 调度任务
```

## 2.3 `MODE` 分派（api 容器如何决定角色）

`api/docker/entrypoint.sh` 依据环境变量 `MODE` 启动不同进程：
- `MODE=migration` → 跑迁移（entrypoint.sh:15）
- `MODE=worker` → `celery -A celery_entrypoint.celery worker ...`（entrypoint.sh:21,66）
- `MODE=beat` → `celery -A app.celery beat`（entrypoint.sh:71）
- `MODE=job` → 一次性运维命令（如 `create-tenant`）（entrypoint.sh:74）
- 默认（无匹配）→ `exec gunicorn ...`（Flask API 服务，entrypoint.sh:126）

## 2.4 一次「知识库问答」请求的组件流经（概览，细节见 03）

1. 浏览器 → nginx → web → api（`/v1/datasets/{id}/retrieve` 或应用 chat 接口）。
2. api 经 `knowledge_retrieval_inner_service` / `retrieval_service` 向**向量库**做向量+关键词检索。
3. 命中分段经 DataPostProcessor（rerank/threshold/metadata 过滤）→ 注入 LLM 上下文。
4. LLM 调用经 **Plugin Daemon**（或内置 provider）完成生成，结果回传 web。
5. 索引阶段（写路径）由 api 写 DB/向量库，并派发 **Celery** 任务给 worker 异步 embedding。

## 2.5 重要架构约束

- `docker-compose.yaml` 是 **generated 文件**（顶部注释：`WARNING: auto-generated by generate_docker_compose. Do not modify directly`）。改动应编辑 `docker-compose-template.yaml` + `docker/.env.example` 后重新生成。
- 本地源码开发**不**用主 compose 跑 api/web，而是用 `docker-compose.middleware.yaml` 只起依赖 + 本地 `uv`/`pnpm` 起服务（见 `04_启动方式报告.md` 方式 B）。
- 向量库、MySQL、各类可选组件通过 **profiles** 启用，未启用则不启动（节省资源）。
